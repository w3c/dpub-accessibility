This document, “Digital Publishing Accessibility: The Digital Publishing Industry's Needs With Respect To W3C/WAI Guidelines” describes how W3C guidelines (including but not limited to WCAG 2.0 [WCAG2], ATAG 2.0 [ATAG2] , UAAG 2.0 [UAAG2] and WAI-ARIA 1.0 [WAI-ARIA]) and their principles, guidelines, and success criteria can be applied to the needs of Digital Publishing. It provides informative guidance, but does not set requirements.

This document is intended to become a Working Group Note.

This is a work in progress. No section should be considered final, and the absence of any content does not imply that such content is out of scope, or may not appear in the future. If you feel something should be covered here, tell us! @@@T.B.D.@@@

Introduction

This document provides informative guidance (but does not set requirements) with regard to the needs of Digital Publishing with regard to W3C guidelines. Is informed by existing guidelines including (but not limited to) WCAG 2.0 [WCAG2], ATAG 2.0 [ATAG2] , UAAG 2.0 [UAAG2] and WAI-ARIA 1.0 [WAI-ARIA]. Where appropriate, it is also informed by other ongoing W3C projects including Portable Web Publications [PWP] and the Web Annotation Working Group.

While the World Wide Web Consortium (W3C)'s W3C Web Accessibility Initiative (WAI) is primarily concerned with web technologies, its guidance is also relevant to technologies which are not exclusively web technologies. The digital publishing industry, which creates mixed web and non-web publications using web-based technological tools, is reliant on the standards set by the W3C.

The current document is focused on the accessibility of digital publishing to people with disabilities and is not intended to supplant any other W3C work.

The W3C and Digital Publishing

"Digital Publishing" is a generic term for the broad ecosystem of electronic journals, magazines, news, or book publishing (authors, creators, publishers, news organizations, booksellers, accessibility and internationalization specialists, etc.). The formats used by eBook readers and tablets for electronic books, magazines, journals and educational resources are largely based on W3C Technologies, such as (X)HTML, CSS, SVG, SMIL, MathML, or various Web APIs. Commercial publishers also rely on W3C technologies in their back-end processing all the way from authoring through to delivering the printed or electronic product and beyond. In general one can say that the Publishing Industry is one of the largest communities relying large palette of W3C technologies.

However, the alignment of the needs of the Publishing Industry and the various W3C recommendations is not satisfactory. Necessary features may be missing in the W3C documents, or may be in draft only; as a result, for example, EPUB3, the standard for electronic books, had to introduce its own extensions to cover the needs of publishing.

Audience

This document is aimed at those involved at creating and modifying existing W3C specifications, especially those oriented around accessibility.

Digital Publishing Issues not currently addressed by W3C Standards

A number of requirements for the digital publishing industry are not currently addressed by W3C standards. Those which can be meaningfully addressed by current WAI efforts are listed here in order of priority. Gaps which have been recognized but cannot be usefully addressed without further research are documented Appendix A. Some of these suggestions request changes to specifications and guidelines. Others request changes to existing techniques documents. Others offer suggestions for further outreach. Where possible, we indicated which set of guidelines might be best positioned to address the change. However, these suggestions are based on the limited knowledge of the Digital Publishing Interest Group, and may be incorrect.

Navigating Among Multiple Documents

Suggestion: Add discussion of packaged documents to WCAG or author extension of WCAG for packaged documents.

Many digital publications are made up of more than one document. For example, often, every chapter in a book be a separate HTML file. In an EPUB file, these are packaged together using a specified mechanism with a method of navigating to the components of the publication. At this point, there is no prescribed method in W3C specifications for navigating among related or packaged components. All documentation and most tools related to accessibility assume that the user is encountering one document.

Use Case: User follows link from index to a paragraph in middle of chapter 23 and realizes that the second entry in index is more helpful. User must return to index (separate HTML document) to access the link to the second index entry.

Use Case: Publisher wants to test accessibility of packaged publication to assess whether users can access all elements listed in the table of contents of a publication. Each HTML file passes validation and is accessible itself, but non-sighted reader cannot access one component. Developers did not catch this because it is technically WCAG-compliant.

Skippability

Suggestion for UAAG: User agents need to implement skippability, and it cannot be confined to users of AT.

Comments: This is a user preference / personalization issue. User must be able to customize what is visible, what is read, etc.

  1. This should be an option for visual as well as aural presentation.
  2. Skipped elements need to be discoverable later.

Use cases:

  • Use Case: User does not want to hear page numbers or headers, but does want to hear footnotes.

References:

  • Daisy Skippable Structures Recommendation

Escapability

Suggestion for UAAG: User agents need to implement escape to the next sibling or parent element, and it cannot be confined to users of AT.

Comments: Users need to be able to immediately move to the next non-contained block element. This is particularly a problem for users who browse the web at high magnification, and for users on mobile. When a user is in the middle of a lengthy element, there needs to be a way for them to move to the next sibling or parent with a simple command. Mobility impaired users are also greatly affected by this need, especially but not exclusively on mobile.

  1. This should be an option for visual as well as aural presentation.

Use cases:

  • Use Case: A chemistry book has a long table of each of the elements in the periodic table. The book's reader does not need to see all of the elements in the table, but they do need to read the text immediately afterwards. They need a simple way to get from the complex table to the next sibling or parent element without endless scrolling.

Page Numbers / Pagination

Suggestion: UAAG suggestion and WCAG suggestion, probably also ATAG suggestion

Comments: For electronic publications, especially those intended to have direct correspondence to printed "hardcopy" publications such as textbooks, a way needs to exist for a user to accurately identify the page number, and to navigate accordingly. This method must be accessible, and should not disrupt continuous reading by screen reader users, and should be accurate and accessible for users using Assistive Technologies which may resize or reflow text on the screen.

A method for providing page numbers is addressed in CSS Paged Media Module Level 3 which applies to the scenario of pagination of text being sent to a printer from a web page. If these or similar methods can be utilized in user-agents to facilitate access to accurate page numbers and pagination, a solution to the above stated issue may already exist.

More investigation and discussion with the pagination CSS work being done is necessary.

References

  • CSS Paged Media Module Level 3 Page-Margin Boxes

Phonetic spellings of proper nouns and jargon terms

Suggestion 1 for UAAG: possible addition to UAAG techniques.

Suggestion 2 possible new or modified specification or guideline for the content itself.

Comments: Publishers need to be able to put in suggested pronunciation for proper nouns, acronyms, and field-specific jargon terms. The Pronunciation Lexicon (PLS) exists but it is exceptionally difficult to use, and doesn't scale well to a large publisher's use cases.

Use cases:

  1. Use Case: A chemistry book needs to use the accepted pronunciations of chemistry terms to maximize clarity. It references a pre-existing pronunciation dictionary made available by the American Chemical Society that uses community-agreed-upon standards.
  2. Use Case: The trade novel THE LIVES OF CHRISTOPHER CHANT by Diana Wynne Jones, in which there is a plot point that a character's name is spelled "Ralph" but pronounced "Rafe," can be correctly encoded using a local pronunciation dictionary.
  3. Use Case: Reading systems support the pronunciation dictionary reference and expose the data to Accessibility APIs.

References:

  • CSS Speech Module
  • Pronunciation Lexicon Specification (PLS)

Semantic List-Head

Suggestion 1: outreach to ARIA WG about proposing aria-labelledby be exposed by user agents without AT.

Suggestion 2: outreach to various groups about alternative non-ARIA semantically meaningful list heading.

Comments: There is no clear method of associating a title of a list with the list items. This can be accomplished using artificial wrappers, such as <div>, but it seems that the list elements (<ul> and <ol>) should include titles. It is tempting to use the h1 – h6 elements, but these have the native semantics of aria role=”heading”, offering unwanted default navigation. It would be useful if there were a role of element called "listtitle" or something similar.

  1. This should be an option for visual as well as aural presentation.

This can be accomplished using aria-labeledby. However, while provide exactly the necessary functionality for screen reader users, it is only exposed to users of AT. Digital publishing needs this association between a list heading in a list to be available to users who are not using AT. This could be accomplished by a non-aria attribute or element, or by accomplished by making aria-labeledby available to non AT.

The following is a simple example: 

<ul>
<listtitle>Noble Gases</listtitle>
<li>Helium</li>
<li>Neon</li>
<li>Argon</li>
<li>Krypton</li>
<li>Xenon</li>
<li>Radon</li></ul>

References:

  • Item #4 5/15/2015 DPUB Minutes

Drop Cap

Suggestion 1 for UAAG: possible addition to UAAG techniques.

Suggestion 2 for WAI Outreach: Outreach to content creators that CSS 3 solves the drop cap problem accessibly.

Comments: The CSS 3 selector first-letter allows the creation of accessible drop caps, and it is widely supported in user agents. However, it is not widely supported in authoring tools, nor is it widely known to content creators. When drop caps are created without using this CSS3 selector, this causes accessibility issues when the two components of the word are read as separate words. Outreach to user agent creators and content creators is needed.

Layout's Influence On Comprehension

Suggestion: Although a solution is needed, the DPUB IG has no good suggestions at this time. More research and investigation is warranted.

Comments: There is currently no reasonable, sustainable method to provide visual information about the layout of text on a page. Certain literary styles such as poetry may use the placement of text to convey extra meaning. There is no consistent method for content creators to make this additional information available to a non-visual display such as a screen reader. We have no suggested approaches at this time. More research and investigation is warranted.

Relevant work being developed outside the W3C

Some open specifications under development or maintained outside the W3C have great relevance to the accessibility in the digital publishing sector. Members of the W3C involved in accessibility within digital publishing need to be aware of work happening in these spaces. Here we describe some key projects that are relevant to our work.

EPUB

EPUB's relevance to the W3C and accessible digital publishing

EPUB is quickly becoming one of the de facto standards in digital publishing, and is gaining global traction. There are a number of mainstream reading systems that can read ePub books. This standard is actively being updated and improved. EPUB is based on HTML5 and adheres to the WCAG accessibility standards.

Maintainers of the specification

IDPF (International Digital Publishing Forum) created and maintains the EPUB® standard.

Contact:

  • Bill McCoy (IDPF)

Historical Context

Beginning with EPUB 3.0, the primary accessible publishing standard, DAISY DTBook and the primary commercial publishing format merged. DAISY’s Digital Talking Book [Z3986-2005] is an XML-based file format and puts an emphasis on structural encoding to enable reading for the visually impaired audience. EPUB 2 allowed DTBook as an alternate syntax. The EPUB 3 specification merged DAISY and EPUB, making accessibility considerations a concern for all publishers, authors, and readers. The EPUB 3.0.1 specification is available at http://idpf.org/epub/301. One of the features of this relationship is the EPUB structural semantics vocabulary, a namespaced vocabulary that enables semantic inflection of HTML elements. In 2015, several of these terms have been transformed into prefixed ARIA roles in Digital Publishing WAI-ARIA Module 1.0.

The IDPF and W3C Digital Publishing Interest Group work closely together on many issues, including broadening the definition of Web Accessibility to include long-form content. The Digital Publishing Interest Group’s Charter includes advancing EPUB 3 as one of the goals. Among the success criteria are providing advice to the IDPF EPUB 3.1 Working Group and advancing what is now called Portable Web Publications, a vision for the convergence of future of digital documents and the Web. Integrating the accessibility functionality that is native to the Web with the additions that DAISY and the IDPF have created is part of that vision.

Diagrammar

Diagrammar's relevance to the W3C and accessible digital publishing

The W3C standards bodies have developed multiple ways of creating extended descriptions for certain page elements, available for accessibility purposes. WAI-ARIA 1.1 has complete when we know the answer, and HTML5 has longdesc. Neither of these fulfills all of the use cases needed by accessible publishing, however. Diagrammar is being developed to address this set of use cases.

The Diagrammar team intends that the XML encoding of Diagrammar data will be transformed to HTML for presentation. Therefore, it will be useful for ongoing communication to continue.

Specification Maintainance

Diagrammar is being developed by The DIAGRAM Center, a Benetech initiative, in partnership with the WGBH National Center for Accessible Media and the US Fund for DAISY.

Contacts:

  • DIAGRAM Center

Historical Context

In the early days of the web, there was recognition that images were an obstacle to persons who are blind and others who could not process the visual information. Today, we find that there are many more types of information objects that are not perceivable to persons with disabilities. Furthermore, publishers and other content creators recognize that alternative mechanisms for conveying information can benefit everybody. These extended descriptions not only pertain to images but any web element that is visually complex and needs to be described in more detail; the DIAGRAM Center hosts use cases for extended description information in multiple formats. Alternative representations may be required, and we need a way to link a specific web element to multiple representations be it an extended description, simplified description, or a tactile or auditory representation of the element.

Diagrammar is an XML framework to hold multiple extended description types, to make a wide range of content types accessible: images, tables, charts, animations, 3D animations etc. Diagrammar defines a data model for extended descriptions and associated metadata. It provides a structured, standard way for extended descriptions data to be modeled, including different modality types for extended descriptions.
